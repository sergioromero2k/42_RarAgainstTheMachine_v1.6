⬡ Permitido / Prohibido

✓ Permitido

  • Usar cualquier librería Python (transformers, langchain, bm25s, chromadb...)
  • Usar AI para entender código o depurar — siempre que lo entiendas
  • Modelos alternativos siempre que también funcione con Qwen3-0.6B
  • Expandir los modelos Pydantic con campos extra
  • Crear múltiples índices para diferentes tipos de ficheros
  • Implementar bonus (embeddings, caching, query expansion, hybrid...)
  • Usar venv / conda para aislamiento de dependencias
  • Usar pytest o unittest para tests locales (no se evalúan)
  • Separar índices por tipo de fichero (.py vs .md)

✗ Prohibido

  • Subir al repo: model weights, datasets grandes, outputs generados
  • Usar AI para generar código que no puedas explicar en evaluación
  • Que el programa crashee con excepciones no manejadas
  • Violar el estándar flake8 (líneas largas, imports desordenados...)
  • Funciones sin type hints — mypy debe pasar sin errores
  • Ignorar el tiempo límite: indexing >5min, respuesta >2s/pregunta
  • README no en inglés
  • Omitir los comandos CLI requeridos (index, search, search_dataset, answer, answer_dataset, evaluate)
  • Chunks de más de 2000 chars sin opción configurable por CLI

⭐ Bonus Part

Los bonus deben estar implementados y funcionando — no solo mencionados en el README. Puedes pedir demostrarlos en la evaluación.

🧠

Semantic Embeddings

sentence-transformers + ChromaDB para búsqueda por similitud semántica. Mayor recall en preguntas conceptuales.

⚡

Hybrid Retrieval

Combinar BM25 (keywords) + embeddings (semántica) con Reciprocal Rank Fusion. Mejor que cada uno por separado.

🔄

Query Expansion

Reformular la query con sinónimos técnicos o usando el LLM para generar variantes. Mejora recall en preguntas ambiguas.

💾

Result Caching

Cache en disco de resultados BM25 por hash de query. Acelera el warm retrieval throughput drásticamente.

🚀

Local LLM via vLLM

Usar el propio vLLM (el repo que indexas) para hacer inferencia. Más rápido que transformers puro en GPU.

💡 Recomendación: empieza por el caching (fácil, gran impacto en velocidad) y luego por los embeddings (mayor impacto en recall).

⬡ Consejos & Recomendaciones

El día 11 es tu brújula

La primera medición de recall es el momento de verdad. Si estás por debajo del 70%, para todo y arregla el chunking antes de seguir. El recall no mejora con el LLM — solo con mejor retrieval.

Chunk size: empieza pequeño

Chunks de 1000 chars suelen dar mejor recall que 2000 porque son más específicos. Luego puedes subir. El chunk más pequeño que contenga la respuesta = el mejor chunk.

Los índices de caracteres son sagrados

El evaluador mide overlap por posición de caracteres. Si tus first/last_character_index no son exactos respecto al fichero original, el recall será 0 aunque recuperes el fichero correcto.

Velocidad sin GPU

Si no tienes GPU, usa max_new_tokens=100-150 para Qwen3-0.6B. Las respuestas cortas y directas cumplen el criterio de calidad y llegan a los 2s. Con 4bit quantization (bitsandbytes) cabe en 4GB RAM.

Preprocesa igual en index y query

El error más común: tokenizar diferente al indexar y al buscar. La función de tokenización debe ser exactamente la misma en ambos sitios. Centralízala en un módulo.

Añade el path al texto del chunk

Prepend "File: vllm/engine/llm.py\n" al texto de cada chunk antes de indexar. BM25 puntuará mejor los chunks cuando la query mencione nombres de ficheros o módulos.

AI: úsala, pero entiéndela

Puedes usar AI para entender BM25, depurar, o generar boilerplate. Pero en la evaluación te pueden preguntar cualquier línea de tu código. Si no puedes explicarla, es un problema.

README desde el día 1

Ve anotando decisiones técnicas mientras las tomas. El README final pedirá "design decisions" y "challenges" — es mucho más fácil escribirlos cuando los recuerdas frescos.
