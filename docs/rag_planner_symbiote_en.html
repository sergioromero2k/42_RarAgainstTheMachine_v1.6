✓ Allowed

  • Use any Python library (transformers, langchain, bm25s, chromadb...)

  • Alternative models as long as Qwen3-0.6B also works

✗ Forbidden

  • Push to repo: model weights, large datasets, generated outputs

  • Violate flake8 standard (long lines, unordered imports...)

  • README not written in English
  • Omit required CLI commands (index, search, search_dataset, answer, answer_dataset, evaluate)
  • Chunks over 2000 chars without configurable CLI argument

⭐ Bonus Part

Bonus features must be implemented and working — not just mentioned in the README. You may be asked to demonstrate them during evaluation.

🧠

Semantic Embeddings

sentence-transformers + ChromaDB for semantic similarity search. Higher recall on conceptual questions.

⚡

Hybrid Retrieval

Combine BM25 (keywords) + embeddings (semantics) with Reciprocal Rank Fusion. Better than either alone.

🔄

Query Expansion

Rewrite the query with technical synonyms or LLM-generated variants. Improves recall on ambiguous questions.

💾

Result Caching

Disk cache of BM25 results by query hash. Dramatically speeds up warm retrieval throughput.

🚀

Local LLM via vLLM

Use vLLM itself (the very repo you indexed) for inference. Faster than pure transformers on GPU.

💡 Recommendation: start with caching (easy, high speed impact) then move to embeddings (highest recall impact).

⬡ Tips & Recommendations

Day 11 is your compass

The first recall measurement is your moment of truth. If you're below 70%, stop everything and fix the chunking before moving on. Recall doesn't improve with the LLM — only with better retrieval.

Chunk size: start small

Chunks of 1000 chars usually give better recall than 2000 because they're more specific. You can always increase later. The smallest chunk that contains the answer = the best chunk.

Character indexes are sacred

The evaluator measures overlap by character position. If your first/last_character_index aren't exact against the original file, recall will be 0 even if you retrieve the right file.

Speed without GPU

No GPU? Use max_new_tokens=100-150 for Qwen3-0.6B. Short, direct answers meet the quality criteria and stay under 2s. With 4bit quantization (bitsandbytes) it fits in 4GB RAM.

Preprocess the same way at index and query time

The most common mistake: tokenizing differently when indexing vs searching. The tokenization function must be exactly the same in both places. Centralize it in one module.

Add the file path to the chunk text

Prepend "File: vllm/engine/llm.py\n" to each chunk before indexing. BM25 will score chunks higher when queries mention file names or module names.

AI: use it, but own it

You can use AI to understand BM25, debug, or generate boilerplate. But during evaluation you can be asked about any line of your code. If you can't explain it, that's a problem.

README from day 1

Write down technical decisions as you make them. The final README will ask for "design decisions" and "challenges" — it's much easier to write them while they're fresh in your memory.
